Authentication and Session Management

The platform uses a token-based authentication flow built on JSON Web Tokens (JWTs).
When a user submits valid credentials to the /auth/login endpoint, the server issues
an access token and a refresh token. The access token is short-lived and expires after
24 hours. The refresh token is long-lived and expires after 30 days. Clients are expected
to store the access token in memory rather than local storage, since local storage is
readable by any script running on the page and is a common target for cross-site
scripting attacks.

When the access token expires, the client calls /auth/refresh with the refresh token
to obtain a new access token without forcing the user to log in again. If the refresh
token itself has expired or has been revoked, the server responds with a 401 status
code and the client must redirect the user to the login page. Refresh tokens are
rotated on every use: each call to /auth/refresh invalidates the old refresh token
and issues a new one. This rotation limits the damage if a refresh token is ever
intercepted, because a stolen token can only be used once before it stops working.

Passwords are never stored in plaintext. The system hashes passwords with bcrypt
using a work factor of 12, which is recomputed periodically as hardware gets faster.
Salting is automatic with bcrypt, so two users with the same password produce different
hashes. Failed login attempts are rate-limited per account and per IP address: after
five failed attempts within ten minutes, the account is locked for fifteen minutes
and the user receives an email notification about the suspicious activity.

Multi-factor authentication is optional but strongly recommended for accounts with
administrative privileges. When enabled, the login flow requires a six-digit
time-based one-time password (TOTP) generated by an authenticator app, in addition
to the username and password. Backup codes are issued at setup time in case the user
loses access to their authenticator device. Each backup code can only be used once
and the user is prompted to regenerate the set once fewer than three remain.

Sessions are tracked server-side in a Redis store keyed by a session identifier
embedded in the JWT claims. This allows administrators to forcibly invalidate a
session — for example, when a user reports a stolen device — without waiting for
the token to expire naturally. The session store also tracks the originating IP
address and user agent string, and the system emails the user whenever a login
occurs from a new device or location they have not used before.

Authorization is handled separately from authentication using role-based access
control. Each user is assigned one or more roles, and each role carries a set of
permissions checked against on every API request. The permission check happens in
middleware before the request reaches the route handler, so unauthorized requests
never touch business logic. Common roles include viewer, editor, and admin, with
admin able to manage other users' roles and viewer restricted to read-only endpoints.

Logout invalidates both the access token's session entry in Redis and the refresh
token in the database, so a logged-out token cannot be replayed even if it has not
yet expired. Clients should also discard any locally cached copy of the tokens on
logout. The combination of short-lived access tokens, rotating refresh tokens, and
server-side session tracking is what allows the system to support both fast,
stateless request validation and the ability to revoke access instantly when needed.
